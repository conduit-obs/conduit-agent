# ADR-0013 — Apache-2.0 license; clean-room implementation

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Tech Lead, Field Engineering Lead, Legal (V0 planning)
- **Context**: Conduit operates in a market with multiple existing Apache-2.0 OpenTelemetry-Collector-wrapping agents. Their source is legally available to copy or fork. Three risks argue against doing so: positioning (a "Conduit is a fork of X" story is one we cannot afford), maintenance independence (forking ties our release pipeline to upstream choices we don't control), and field credibility (customers running `git log` on Conduit and finding contributors from a vendor agent will ask questions we don't want to answer in a sales conversation). Independently, the Apache-2.0 license is the standard for OpenTelemetry-ecosystem projects and aligns with field-buyer expectations.
- **Decision**: Conduit is licensed under Apache-2.0. The implementation is clean-room: patterns from other agents may be borrowed conceptually with attribution, but **no code is copied verbatim, even from compatibly-licensed sources**. Any pull request with verbatim copy is rejected and rewritten.
- **Alternatives considered**:
  - Fork an existing OTel agent (rejected: positioning, maintenance, and field-credibility risks).
  - Copy code from a compatibly-licensed agent with attribution (rejected: same risks; Apache-2.0 permits it but the project discipline does not).
  - Pick a different OSS license (rejected: Apache-2.0 is standard for OpenTelemetry-ecosystem projects and aligns with field-buyer expectations).
- **Consequences**: Implementation is slower than copy-and-modify. Positioning and maintenance independence are stronger. Reference materials are tracked in internal planning docs only and never embedded in the public source tree.
