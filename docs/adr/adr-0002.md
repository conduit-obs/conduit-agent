# ADR-0002 — V0 is agent-only; gateway is a destination, not a Conduit role

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Tech Lead, Field Engineering Lead, PM (V0 planning)
- **Context**: Enterprise topologies often have an agent fleet feeding a gateway tier (Refinery, S3 archival, multiple destinations). A "Conduit-as-gateway" role would expand Conduit's addressable role; it would also add stateful processing, multi-destination routing fanout, mTLS bootstrap UX, and HA/sizing guidance. Honeycomb already ships the Honeycomb Collector as a gateway, and any OTLP-capable gateway can serve the role.
- **Decision**: V0 does not ship a Conduit-as-gateway role. Customers who need a gateway tier run the Honeycomb Collector or any OTLP-capable gateway. Conduit emits to one with a single config switch (`output.mode: gateway`). Conduit-as-gateway is V2+.
- **Alternatives considered**:
  - One binary, two roles (`conduit.role: agent | gateway`) (rejected: V2+ — see V2-1 in `conduit-agent-plan/10-roadmap-control-plane-later.md`).
  - Agent-side multi-destination fanout via `routingconnector` (rejected: belongs at the gateway tier).
- **Consequences**: V0 surface stays small. Enterprise topologies (agents → gateway → Refinery / S3 / Honeycomb) work via the customer's gateway tier, not via Conduit. We may face "Conduit vs Honeycomb Collector" confusion; we mitigate via positioning docs and the demo's "switch to gateway" moment.
