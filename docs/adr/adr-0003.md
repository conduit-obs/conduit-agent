# ADR-0003 — Build via OpenTelemetry Collector Builder

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Tech Lead (V0 planning)
- **Context**: Conduit must ship a curated OpenTelemetry Collector binary. Two non-OCB paths exist: forking the upstream collector (catastrophic maintenance burden) or hand-maintaining a `components.go` manifest (also high burden, and loses the upstream tooling). The OpenTelemetry Collector Builder (OCB) is the upstream-supported way to assemble a curated collector.
- **Decision**: Conduit's binary is built via OCB from a curated `builder-config.yaml` checked into the repository root. The build pipeline rewrites OCB output module paths and folds the result into `internal/collector/` so `go build ./...` works end-to-end.
- **Alternatives considered**:
  - Hand-maintained `internal/collector/components.go` (rejected: untested with the upstream patterns, loses tooling).
  - Fork the upstream collector repository (rejected: catastrophic maintenance burden).
- **Consequences**: Each Conduit MINOR pins to one upstream OTel Collector MINOR (per [ADR-0014](adr-0014.md)). Makefile complexity around OCB output is accepted. We benefit from upstream component releases automatically.
