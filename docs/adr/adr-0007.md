# ADR-0007 — Honeycomb-optimized defaults, vendor-neutral schema

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: PM, Tech Lead (V0 planning)
- **Context**: Conduit's strategic purpose is closing the Honeycomb adoption familiarity gap. Tuning defaults for Honeycomb (RED dimension shape, log redaction patterns, resource attribute coercion, default endpoint `api.honeycomb.io`) maximizes the day-one value. At the same time, Apache-2.0 obligates real optionality: the off-ramp story is one of the biggest buyer-side concerns.
- **Decision**: Conduit's defaults are tuned for Honeycomb. The config schema is **vendor-neutral**: `output.mode: honeycomb | gateway`. No required field is named after Honeycomb. Honeycomb-specific concerns (auth header, EU endpoint) live under `output.honeycomb.*`; gateway-specific concerns under `output.gateway.*`.
- **Alternatives considered**:
  - Vendor-named required fields (`honeycomb_url`, `token`) (rejected: schema-level lock-in, breaks the off-ramp story).
  - Fully Honeycomb-only schema (rejected: violates Apache-2.0 spirit and the buyer-side off-ramp narrative).
- **Consequences**: Adding a new top-level mode (e.g., a vendor-specific mode in V1+) is a non-breaking schema addition. Switching destinations is a one-field config change with no rebuild — the operational proof of the off-ramp story.
