# ADR-0008 — `output:` block uses a single `output.mode` field with mode-specific sub-blocks

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Tech Lead, PM (V0 planning)
- **Context**: The destination story is the most important configuration surface. A flat per-signal model (`output.traces.endpoint`, `output.metrics.endpoint`, `output.logs.endpoint`) is more flexible but easy to footgun (e.g., partial misconfiguration that silently drops one signal). A mode-field shape keeps guardrails tight and validation simple.
- **Decision**: Destinations are configured via a single `output.mode` enum (`honeycomb | gateway`) with per-mode sub-blocks (`output.honeycomb.*`, `output.gateway.*`). Refinery is **not** its own mode (see [ADR-0010](adr-0010.md)).
- **Alternatives considered**:
  - Per-signal endpoint configuration (rejected for V0; defer to V1+ if customer demand emerges).
  - Multiple top-level destination blocks for fanout (rejected: V0 does not fan out from the agent — see [ADR-0002](adr-0002.md)).
- **Consequences**: Adding a new top-level mode in a future MINOR is a clean schema addition. Per-signal flexibility comes only via mode-specific sub-blocks (e.g., the Refinery sub-field under Honeycomb). Validation can confidently refuse mismatched sub-blocks (e.g., `output.gateway.*` set when `output.mode: honeycomb`).
