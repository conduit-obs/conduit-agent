# ADR-0004 — Pure upstream OTel components in V0; zero custom processors / receivers

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Tech Lead, PM (V0 planning)
- **Context**: Custom collector components (processors, receivers) drive maintenance debt, fragment the binary from upstream, and risk Conduit's "open and inspectable" positioning. Every behavior V0 needs (RED metrics, k8s enrichment, redaction, resource detection, transformation) is already covered by upstream OTel collector and contrib.
- **Decision**: V0's component manifest pulls only from `go.opentelemetry.io/collector` and `github.com/open-telemetry/opentelemetry-collector-contrib`. **No custom Conduit processors or receivers in V0.**
- **Alternatives considered**:
  - Custom Honeycomb-shaping processor (rejected: "open and inspectable" wins over "convenience").
  - Custom k8s attributes processor (rejected: upstream's `k8sattributesprocessor` is sufficient).
  - Custom Conduit heartbeat receiver (rejected: control-plane primitive, V3).
- **Consequences**: Some Honeycomb-specific shaping (e.g., `team` attribute mapping) lives in user-provided `transformprocessor` blocks rather than in a Conduit-built processor. Upgrades become easier because Conduit moves with upstream rather than diverging.
