# ADR-0010 — Refinery is a sub-field of Honeycomb, not its own output mode

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Tech Lead, Field Engineering Lead (V0 planning)
- **Context**: Refinery is Honeycomb's tail-sampling product. In a Honeycomb deployment that uses Refinery, traces ingress through Refinery while metrics and logs go directly to Honeycomb. Refinery is therefore *how Honeycomb traces ingress when sampling is required*, not a peer destination.
- **Decision**: Trace routing through Refinery is configured via `output.honeycomb.traces.via_refinery.endpoint` — a sub-field under the Honeycomb mode — not as its own `output.mode: refinery`. The V0 mode list stays at two values (`honeycomb`, `gateway`).
- **Alternatives considered**:
  - Own mode `output.mode: refinery_and_honeycomb` (rejected: implies Refinery is a peer destination, which it isn't).
  - Peer-destination model with separate top-level fields (same rejection).
- **Consequences**: Customers cannot configure "send everything to a non-Honeycomb backend through Refinery" — an unusual deployment we deliberately do not support in V0. Customers can configure "Honeycomb for everything, but route traces via our Refinery." Topology 2 in the architecture doc reflects this.
