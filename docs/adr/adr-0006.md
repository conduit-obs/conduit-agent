# ADR-0006 — Allowlist-based RED dimension model with denylist enforcement

- **Status**: Implemented (V0). Original decision date 2026-05-01; M8 implementation landed 2026-05-01.
- **Date**: 2026-05-01
- **Deciders**: Tech Lead, PM (V0 planning)
- **Context**: Cardinality-driven cost spikes are the single most common metrics-bill failure mode. Customers do not always know which span attributes are dangerous as metric dimensions until the bill arrives. Conduit must protect customers by default.
- **Decision**: Default RED dimensions are an explicit allowlist of safe upstream-OTel attributes (`service.name`, `deployment.environment`, `http.route`, `http.method`, `http.status_code`, `rpc.system`, `rpc.service`, `rpc.method`, `messaging.system`, `messaging.operation`, `k8s.namespace.name`, `cloud.region`, `team`). High-cardinality attributes are explicitly denylisted at schema validation: `trace_id`, `span_id`, `request_id`, `user.id`, `customer_id`, `tenant_id`, `url.full`, `http.url`, `http.path`, `http.target`. A connector-level cardinality cap (`aggregation_cardinality_limit`, default 5000) bounds the connector's working set; combinations beyond the cap roll into a single `otel.metric.overflow="true"` overflow series, which `conduit doctor` reads to emit CDT0510.
- **Alternatives considered**:
  - Allow-everything-by-default with cardinality cap only (rejected: too easy to footgun).
  - Minimal default with no allowlist customization (rejected: too rigid; customers need to add dimensions like `team`).
  - Custom in-process cardinality observer running independently of the connector (rejected during implementation: the upstream connector's `aggregation_cardinality_limit` + overflow series provides the same signal as a first-class feature, and a custom observer would have duplicated state. The doctor (M11) reads the existing overflow metric instead of running its own counter).
- **Consequences**: Customers must explicitly opt in to add a dimension that the denylist forbids; doing so fails `Validate` at parse time with an error citing CDT0501 + this ADR. The cardinality cap moves cost from "OOM" to "a single visible metric" so cardinality drift becomes a query-time observation rather than a crash. Promoting a denylisted attribute to allowed (e.g. someone wants `customer_id` for a known-bounded customer set) requires an ADR amendment, not a config change.
- **Implementation**:
  - Schema: `internal/config/types.go` (`REDConfig`, `REDDefaultSpanDimensions`, `REDDefaultResourceDimensions`, `REDDefaultHistogramBuckets`, `REDDimensionDenylist`).
  - Validation: `internal/config/validate.go` `validateMetrics`.
  - Rendering: `internal/expander/expander.go` `applyREDView` + `internal/expander/templates/base.yaml.tmpl` `connectors:` block.
  - Operator surface: `cmd/preview/preview.go` `printREDProjection` writes the projected dimension set + cap to stderr alongside the rendered YAML.
