# ADR-0006 — Allowlist-based RED dimension model with denylist enforcement

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Tech Lead, PM (V0 planning)
- **Context**: Cardinality-driven cost spikes are the single most common metrics-bill failure mode. Customers do not always know which span attributes are dangerous as metric dimensions until the bill arrives. Conduit must protect customers by default.
- **Decision**: Default RED dimensions are an explicit allowlist of safe upstream-OTel attributes (`service.name`, `deployment.environment`, `http.route`, `http.method`, `http.status_code`, `rpc.system`, `rpc.service`, `rpc.method`, `messaging.system`, `messaging.operation`, `k8s.namespace.name`, `cloud.region`). High-cardinality attributes are explicitly denylisted at schema validation: `trace_id`, `span_id`, `request_id`, `user.id`, `customer_id`, `tenant_id`, `url.full`, `http.url`, raw path. A runtime cardinality observer warns when any dimension exceeds a configurable threshold (default 1000).
- **Alternatives considered**:
  - Allow-everything-by-default with cardinality cap only (rejected: too easy to footgun).
  - Minimal default with no allowlist customization (rejected: too rigid; customers need to add dimensions like `team`).
- **Consequences**: Customers must explicitly opt in to add a dimension that the denylist forbids; doing so emits a `conduit doctor` warning that names the attribute and links to docs. The runtime observer adds a small overhead per signal but pays for itself when it prevents a cardinality blow-up in production.
