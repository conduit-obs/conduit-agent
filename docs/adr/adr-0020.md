# ADR-0020 — Zero-code application instrumentation via OBI as a Collector receiver

- **Status**: Accepted
- **Date**: 2026-05-03
- **Deciders**: Tech Lead (5 design questions answered 2026-05-03)
- **Context**: V0 ships host telemetry (hostmetrics, system logs) and a pass-through OTLP receiver for SDK-instrumented application traces. The gap is application observability for services *without* SDK instrumentation — the case where an operator inherits a fleet they did not write the code for. The [OpenTelemetry eBPF Instrumentation project (OBI)](https://opentelemetry.io/docs/zero-code/obi/) closes that gap as an upstream OTel component: an eBPF auto-instrumentation agent that captures HTTP / gRPC / JSON-RPC / database RED metrics and distributed trace spans without code changes, and ships as a Collector receiver (Go module `go.opentelemetry.io/obi`, import path `go.opentelemetry.io/obi/collector`) that produces OTLP-shaped data into the existing pipeline. Four constraints shape the integration. (1) OBI is Linux-only (kernel 5.8+ or RHEL-family 4.18+ with backports), `amd64` / `arm64`, and requires either root or a specific set of eBPF capabilities — upstream documents the full set as `CAP_SYS_ADMIN`, `CAP_DAC_READ_SEARCH`, `CAP_NET_RAW`, `CAP_SYS_PTRACE`, `CAP_PERFMON`, `CAP_BPF` (plus `CAP_CHECKPOINT_RESTORE` in their k8s example). (2) OBI emits its own RED metrics, which collide with M8's `span_metrics` connector if both run unfiltered against the same dataset. (3) The receiver carries non-trivial weight (eBPF tooling, libbpf bindings, ~30–60 MB additional binary size) and ADR-0004 forbids custom processors / receivers in V0, so the integration must live entirely in upstream OTel components. (4) As of OBI v0.8.0, the published Go module **does not** ship the generated eBPF Go bindings — building any consumer of `go.opentelemetry.io/obi/collector` requires checking out the OBI source, running `make docker-generate` (or `make generate` with a local C/clang toolchain), and pointing OCB at the local copy via a `replaces:` directive in `builder-config.yaml`. This is upstream's design, not a Conduit constraint, and is expected to evolve.
- **Decision**: Conduit V0.1 adds OBI as an opt-in zero-code application instrumentation path, exposed in `conduit.yaml` under a top-level `obi:` block, gated behind a single `enabled` flag with profile-specific defaults. The integration is built from five sub-decisions, each captured below.

  1. **Build shape — single Linux binary, OBI compiled in unconditionally (when shipped).** The `obireceiver` component is added to `builder-config.yaml` (the OCB manifest at the repo root, source of truth for `internal/collector/components.go`) under a Linux-only stanza; the Windows and macOS Conduit binaries do not carry it. There is no separate `conduit-obi` Linux variant; one Linux binary serves both OBI and non-OBI deployments. Cost: every Linux operator carries the eBPF tooling weight whether or not they enable OBI. Benefit: one .deb, one .rpm, one container image, no fork in the install / Helm / runbook flows. **Implementation note (V0.1):** the OCB manifest change is staged behind a separate decision because the BPF-binding generation requirement (see Context constraint 4) means adding OBI to the build pipeline is not a manifest-only change — it requires either a CI codegen step (clone OBI → `make docker-generate` → `replaces:`) or a vendored fork. The schema, expander, doctor, install script, and Helm chart all land in V0.1 with the OBI surface ready; the OCB manifest line and the integration test follow once the build-pipeline approach is decided. With OBI not yet linked, `conduit doctor` reports a clear FAIL when `obi.enabled: true` so operators get an actionable error rather than a silent startup failure.

  2. **Privilege escalation — explicit, opt-in, idempotent.** When the operator passes `--with-obi` to `scripts/install_linux.sh`, the script writes a systemd drop-in (`/etc/systemd/system/conduit.service.d/obi.conf`) granting the upstream-documented capability set (`CAP_SYS_ADMIN`, `CAP_DAC_READ_SEARCH`, `CAP_NET_RAW`, `CAP_SYS_PTRACE`, `CAP_PERFMON`, `CAP_BPF`) via both `CapabilityBoundingSet=` and `AmbientCapabilities=`, then reloads the unit. Re-running the installer is safe (drop-in is regenerated, not appended). The Helm chart adds an `obi.enabled` value; when set to `true`, the daemonset's `securityContext.capabilities.add` includes the same set plus `CHECKPOINT_RESTORE` (per the upstream k8s example). Conduit does **not** auto-escalate caps based on `conduit.yaml` content alone — the privilege grant is always an explicit operator action visible in the install command or Helm values.

  3. **RED metric collision — scope-tag by default, opt-in connector suppression.** When OBI is enabled, every metric it emits is annotated with `instrumentation.scope = "obi"` (this is OBI's native behavior; we rely on it rather than reproducing it in a transform processor). Operators filter at query time when they want one source or the other. For operators who want OBI to be the exclusive RED source, `obi.replace_span_metrics_connector: true` causes the expander to suppress the M8 `span_metrics` connector emission entirely. Default is `false` (both sources flow; `instrumentation.scope` disambiguates). The dual-source default preserves continuity with M8 and avoids surprise behavior changes when an operator flips OBI on partway through a deployment.

  4. **Profile defaults — off everywhere except k8s.** `obi.enabled` defaults to `false` on the `linux`, `darwin`, `windows`, and `docker` profiles. On the `k8s` profile, `obi.enabled` defaults to `true`. Rationale: OBI's auto-discovery is dramatically more useful in Kubernetes (it leverages the k8s informer to map processes to workloads), and "zero-code application observability" is the primary reason most operators reach for an agent in a cluster. Everywhere else the defaults stay conservative because the privilege grant is an operator decision, not a platform decision. The Helm chart documentation calls out the k8s default-on prominently so it is not a surprise.

  5. **Escape hatch — existing `overrides:`, no new mechanism, minimal curated schema.** ADR-0012 establishes `overrides:` as the only escape hatch from the curated `conduit.yaml` schema into raw upstream collector YAML. OBI follows the same rule. The V0.1 curated schema exposes exactly two knobs — `enabled` and `replace_span_metrics_connector` — because (a) OBI's auto-discovery already handles the common case without configuration, and (b) ADR-0012 says "promote knobs when patterns emerge," so we start minimal. Anything else (instrumentation targets, feature flags, k8s metadata enrichment, discovery poll interval) goes through `overrides.receivers.obi.<...>`, which the embedded collector resolver deep-merges at startup. No `obi.extra:` field, no per-field denylist on the OBI sub-tree of `overrides:`. When operators reach for `overrides:` to tune OBI heavily, that is a signal to promote the field to the curated schema in a future release, per ADR-0012's stated review process. (An earlier draft of this ADR named `discovery.services`, `routes.patterns`, and `features` as additional curated fields; those were removed when implementation showed they didn't all map to real OBI v0.8.0 schema fields and the user's spirit of "simple yaml flag... lets me overwrite and nest configurations" was already covered by the two-knob + `overrides:` shape.)

- **Alternatives considered**:
  - **Two Linux binary variants (`conduit` lean and `conduit-obi`).** Rejected for V0 because the doubled release matrix (debs / rpms / images / Helm charts) costs more in maintenance than the 30–60 MB saves in deployments where eBPF is not needed. Reconsider if binary size becomes a real customer complaint.
  - **OBI as a sidecar process invoked by Conduit.** Rejected because OBI's first-class integration path is already the Collector receiver, and a sidecar would duplicate process supervision, OTLP wire serialization, and resource accounting that the collector already does. The receiver pattern also means OBI's output flows through the existing `resource_detection`, `transform`, and `batch` processors automatically.
  - **Auto-escalating caps based on `conduit.yaml`.** Rejected because cap changes require a service restart, and "edit YAML, get an unexpected privilege grant on next config reload" is the wrong shape for a security-relevant change. Explicit `--with-obi` at install time is auditable and obvious.
  - **`obi.exclusive_red` boolean replacing the connector toggle.** Rejected for naming clarity — `replace_span_metrics_connector` reveals the underlying mechanism (which `conduit preview` shows anyway) and makes the toggle's effect predictable for operators who already know the M8 RED path.
  - **`obi.extra:` block as a per-receiver escape hatch.** Rejected because it would be a second escape hatch, directly conflicting with ADR-0012. The existing `overrides:` mechanism already covers the use case.
- **Consequences**:
  - **OCB manifest grows (deferred).** A future PR adds `go.opentelemetry.io/obi` to `builder-config.yaml`'s `receivers:` list under a Linux-only build path, with the corresponding `replaces:` directive pointing at a vendored or CI-generated OBI checkout (see "Open question: build pipeline" below). Per [ADR-0014](adr-0014.md), the OBI version pin moves on the same monthly cadence as the rest of the contrib pin.
  - **Schema work (V0.1).** `internal/config/types.go` gains an `OBI` struct with two fields: `enabled` (`*bool` so omitted vs. explicit-false are distinguishable for profile defaulting) and `replace_span_metrics_connector` (`bool`, default false). The validator rejects `obi.enabled: true` on non-Linux profiles (`darwin`, `windows`, and `none` — `auto` is resolved at expand time) with a remediation message that names the current profile. The validator also rejects `obi.replace_span_metrics_connector: true` when `obi.enabled: false` so the configuration matches its effect.
  - **Defaults work (V0.1).** `applyDefaults` in `internal/config/load.go` sets `obi.enabled = true` on the `k8s` profile when the operator omitted the block, and leaves it `false` on every other profile.
  - **Expander work (V0.1).** `internal/expander/templates/base.yaml.tmpl` gains a conditional `receivers.obi:` block and pipeline plumbing that adds `obi` to the `traces` and `metrics` pipeline `receivers:` lists when `obi.enabled` resolves to true. When `obi.replace_span_metrics_connector: true`, the `connectors:` block and the `span_metrics` exporter teeing on the traces pipeline are suppressed. The k8s default-on flows through profile defaulting, not a profile YAML fragment, so the rendered output remains a single block. All 10 goldens regenerate; the `06-k8s-default` golden gains the OBI receiver block and pipeline membership.
  - **Doctor work (V0.1).** `conduit doctor` adds an OBI preflight (`CDT0204 — receiver.obi`) that checks: (a) the OBI receiver factory is registered in the running binary (catches "you have OBI on but built without it"); (b) kernel version (≥ 5.8 or RHEL-family ≥ 4.18 with backports); (c) BTF availability (`/sys/kernel/btf/vmlinux`); (d) the required ambient/effective caps on the running process; (e) `/proc/sys/kernel/unprivileged_bpf_disabled`. When `obi.enabled: true` and any preflight fails, doctor exits non-zero with the exact remediation line. The check is `SeveritySkip` when `obi.enabled` is false, so non-OBI installs are not bothered.
  - **Install / Helm work (V0.1).** `scripts/install_linux.sh` gains `--with-obi`. The systemd drop-in lives at `/etc/systemd/system/conduit.service.d/obi.conf`. `deploy/helm/conduit-agent/values.yaml` gains `obi.enabled` and the daemonset template adds a `securityContext.capabilities.add` block + `hostPID: true` when `obi.enabled` is true.
  - **Integration test (deferred).** A testcontainers-go test boots a privileged Debian container, installs the .deb, sets `obi.enabled: true`, runs a target HTTP server inside the container, generates traffic, and asserts that RED metrics with `instrumentation.scope = "obi"` arrive at a sidecar OTLP receiver. Deferred until the OCB manifest update lands (the test needs OBI in the binary to be meaningful).
  - **Documentation work (V0.1).** `README.md`, the `docs/getting-started/*.md` pages for linux + k8s, and a new `docs/getting-started/obi.md` cover the privilege grant, the dual-RED-source default, and the kernel requirements. The k8s page leads with OBI default-on as a prominent callout.
  - **Operational risk: privilege footgun.** Granting `CAP_BPF` etc. to the `conduit` user materially expands its blast radius on the host. The sub-decision 2 split — separate explicit `--with-obi` at install time, separate explicit `obi.enabled` at runtime — is intentional defense in depth. A compromised `conduit.yaml` cannot grant OBI caps; a compromised installer flag set still requires `obi.enabled` to actually use them.
  - **Operational risk: kernel compatibility.** RHEL 7 and pre-5.8 Debian / Ubuntu LTS (notably Ubuntu 18.04, EOL) cannot run OBI. The doctor preflight catches this loudly; the install script logs a warning when `--with-obi` runs against an incompatible kernel and refuses to write the drop-in.

- **Build pipeline (resolved 2026-05-05).** ~~Adding `go.opentelemetry.io/obi` to `builder-config.yaml` requires that OCB resolve the BPF Go bindings, which OBI v0.8.0 does not ship in its published module. The three viable approaches are (a) vendor OBI as a git submodule and run `make docker-generate` against it in CI, (b) maintain a Conduit fork of OBI with bindings committed and bump it on the same cadence as upstream, (c) wait for upstream OBI to ship pre-generated bindings (open issue, no timeline).~~ Approach **(a) clone-and-codegen-in-CI** is now in place. The integration shape is:

  - `internal/collector/components_obi_linux.go` (`//go:build linux`) is the single point in the agent that imports `go.opentelemetry.io/obi/collector`. It registers the OBI receiver factory via an `addPlatformReceivers` hook called from the OCB-generated `components.go`. A matching `components_obi_other.go` (`//go:build !linux`) provides a no-op stub so macOS / Windows builds compile cleanly without resolving the OBI module.
  - Committed `go.mod` carries `require go.opentelemetry.io/obi v0.8.0` (forced in by `go mod tidy` walking the Linux-tagged file) but **does not** carry a `replace` directive. The upstream proxy v0.8.0 is sufficient to satisfy module resolution on macOS / Windows, where no Go file ever compiles against the OBI module.
  - `make obi-vendor` (a) clones the OBI source into the gitignored `third_party/obi/`, (b) runs upstream's `make docker-generate` to produce the eBPF Go bindings, and (c) injects `replace go.opentelemetry.io/obi => ./third_party/obi` into `go.mod` via `go mod edit`. `make obi-clean` reverses (c). The replace directive only ever lives in working trees / CI runners, never in commits.
  - Every Linux job in `.github/workflows/ci.yml`, `release.yml`, and `codeql.yml` runs `make obi-vendor` before any Go toolchain step. Linux release artifacts (goreleaser) and the conduit-agent container image link OBI in unconditionally; macOS / Windows artifacts skip OBI via the build tag.
  - **Trade-off accepted:** every Linux CI run pays ~2–3 min for the upstream clone + `make docker-generate` step (Docker-bound). Caching the `third_party/obi/` tree as a GitHub Actions artifact is a future optimization. The "always pull latest at the pinned tag, accept that an upstream break breaks our build" stance is intentional — OBI's BPF surface evolves quickly and a stale fork would mask compatibility regressions until release time.

  Approaches (b) and (c) remain on the table if upstream's pace makes (a) too noisy: vendoring a Conduit-maintained OBI fork would trade upstream-currency for build stability, and waiting for upstream to ship pre-generated bindings would obviate the codegen step entirely.

- **Amendment (2026-05-05) — language injectors promoted to curated knobs; context propagation stays under `overrides:`.** End-to-end testing against the Java meminator workshop on a kind cluster surfaced two patterns that were inevitable but underappreciated when the curated schema was first scoped to two fields. (1) Pure-eBPF instrumentation of TLS-internal Java services emits decorrelated spans because the eBPF tracer can only see encrypted bytes on the SSL/TLS write path; the only way to inject the W3C `traceparent` header on those calls is OBI's bytecode-injecting Java agent, gated behind `receivers.obi.javaagent.enabled`. (2) NodeJS services have the same break, less because of TLS and more because libuv's vectored writes don't fit the eBPF tracer's reconstruction patterns; OBI exposes a parallel `nodejs.enabled` switch that loads its language-side hook. Reaching for those switches via `overrides.receivers.obi.javaagent.enabled` is two levels deep and the most-asked enterprise knob, so per ADR-0012's "promote when patterns emerge" review process, both are now first-class curated fields:
  - `obi.java_tls` (`*bool`, default `false`) → `receivers.obi.javaagent.enabled`
  - `obi.nodejs` (`*bool`, default `false`) → `receivers.obi.nodejs.enabled`

  Both pointer-bools so applyDefaults can keep them off everywhere, both Linux-only by transitivity (validation already rejects `obi.enabled: true` off Linux, and the cross-field rule rejects `java_tls: true` while `enabled: false`). Doctor's CDT0204 gains a Java preflight that warns when `obi.java_tls: true` but no JVM is visible in `/proc`. The Helm chart exposes both as `obi.javaTLS` / `obi.nodeJS` values.

  Cross-service eBPF context propagation (`receivers.obi.ebpf.context_propagation: all` plus `track_request_headers: true`, which uses `bpf_probe_write_user` to inject the `traceparent` byte sequence into outbound HTTP/gRPC writes at the syscall layer) deliberately stays under the `overrides:` escape hatch rather than getting promoted. Two reasons: (a) it requires kernels with `bpf_probe_write_user` available (not all hardened kernels expose it; some distros disable it via `CONFIG_BPF_KPROBE_OVERRIDE=n`), and (b) it adds an audit-log line every time the kernel writes user-space memory, which security teams notice. Operators who need it use:

  ```yaml
  overrides:
    receivers:
      obi:
        ebpf:
          context_propagation: all
          track_request_headers: true
  ```

  And the chart surfaces this via the new `conduitOverrides:` value that splices a verbatim `overrides:` block into the rendered conduit.yaml. Promotion to a curated knob is open for a future amendment if the propagation toggle becomes the common path.

  Two adjacent fixes landed in the same change because the testing also caught them: (1) the rendered `receivers.obi:` block now emits `discovery.instrument: [{open_ports: 1-65535}]` because `meter_provider.features` is a downstream metric-export filter, not the on-switch for `FeatureAppO11y`, and OBI rejects the config at startup ("at least one of 'network', 'application' or 'stats' features must be enabled") without an explicit discovery selector; (2) the Helm ClusterRole now includes the core API group's `services` resource because OBI's k8s informer caches Service objects (the reflector logged `services is forbidden` and would have looped indefinitely). Both are pure correctness fixes — no schema change, no operator-visible behavior beyond "the agent now starts and OBI traces appear."
