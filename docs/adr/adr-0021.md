# ADR-0021 — Profile-default `service.name` and resource-processor `insert` semantics

- **Status**: Accepted
- **Date**: 2026-05-04
- **Deciders**: Tech Lead (3 design questions answered 2026-05-04)
- **Context**: Conduit V0 made `service_name` a hard-required field in `conduit.yaml`. Operators installing the agent on a Linux host with no `--service-name` flag fell back to `hostname -s` (e.g. `prod-web-12.us-east-1.compute.internal`), producing one Honeycomb dataset per host and making fleet-wide boards effectively impossible to target. Two consequences flowed from this. (1) The checked-in dashboards under `dashboards/*.json` had to hardcode an ad-hoc dataset name (`prod-infra`) that no actual install reproduces, and the M11 `conduit board apply` work assumed every operator would post-process boards before applying. (2) The expander's `resource:` processor used `action: upsert` for `service.name` on every signal, which silently rewrites the `service.name` of incoming OTLP traffic from peer applications — an app pod sending traces with `service.name=checkout` has them relabeled to whatever the agent's `service_name` is. The two problems are adjacent: a useful default for the agent's own identity only works if it doesn't clobber forwarded traffic, and the upsert-clobber is much more visible the moment defaults make agent-shaped service names common.
- **Decision**: Conduit V0.x adopts (a) profile-shaped default `service.name` values applied in `applyDefaults()` when the operator omits the field, and (b) `action: insert` (not `upsert`) for `service.name` on the resource processor. `deployment.environment` keeps its existing `action: upsert` semantics. The combined change is captured in three sub-decisions.

  1. **Profile-shaped defaults, not hostname.** When `service_name` is empty after YAML decode and the resolved profile platform is one of `linux`, `darwin`, `windows`, `docker`, or `k8s`, `applyDefaults()` fills it in with the platform-shaped default per the table below. `host.name` carries the per-host slice within the dataset (already populated by `resourcedetection`), so a 50-host fleet on the linux profile flows into one `linux-host` dataset that boards can target, with `host.name` available for break-down and the Host preset filter. Operators override by setting `service_name:` in `conduit.yaml`, by passing `--service-name=` to the installer, or by setting `conduit.serviceName` in the Helm chart values — the explicit value always wins over the profile default.

     | Profile | Default `service.name` |
     |---|---|
     | `linux`   | `linux-host`   |
     | `darwin`  | `macos-host`   |
     | `windows` | `windows-host` |
     | `docker`  | `docker-host`  |
     | `k8s`     | `k8s-cluster`  |
     | `none` / unresolvable `auto` | (unchanged: `service_name` remains required) |

     `mode: none` keeps the existing required-field behavior because OTLP-only installs have no host narrative to default to and the operator is explicitly opting out of the platform shape.

  2. **`action: insert` for `service.name`.** The resource processor in `internal/expander/templates/base.yaml.tmpl` switches `service.name` from `action: upsert` to `action: insert`. With `insert` semantics, the agent's `service.name` is applied only to signals that arrive without one — host-emitted signals (hostmetrics, journald, filelog) get the agent's profile-shaped default, while OTLP traffic forwarded from peer applications retains the application's own `service.name`. This is the correct behavior for the gateway use case (an app sending `service.name=checkout` keeps that on the wire) and for the host use case (the agent's own host metrics get `service.name=linux-host`). `deployment.environment` keeps `action: upsert` because environment tagging is something the operator deliberately wants to win over whatever the application emitted — environment promotion is a deployment concern, not an application concern.

  3. **Boards target profile-shaped datasets.** The five checked-in boards under `dashboards/` (`linux-host-overview.json`, `macos-host-overview.json`, `windows-host-overview.json`, `docker-host-overview.json`, `k8s-cluster-overview.json`) update their log-panel `dataset:` fields from the placeholder `prod-infra` to the matching profile default (`linux-host`, `macos-host`, etc.). Metric panels continue to query the `metrics` dataset, which Honeycomb routes by metric type rather than service name. The `dashboards/README.md` documents the dataset assumption so future board PRs match.

- **Alternatives considered**:
  - **Resolve `${env:CONDUIT_SERVICE_NAME}` in conduit's Go config layer to enable applyDefaults to see through env-var indirection.** Rejected because adding env-var resolution to the conduit Go layer duplicates work the embedded Collector already does at startup, and the alternative — drop the env-var indirection from the shipped `conduit.yaml.default` files — is simpler and forces the new install path to flow through the literal-write code path that's easier to debug. Operators who deliberately want env-var indirection re-add the line by hand; the comment block in the shipped default documents how.
  - **`conduit-linux` / `conduit-macos` / ... prefix instead of profile-host suffix.** Rejected because the `linux-host` shape reads more naturally in board titles ("Linux Host Overview" → dataset `linux-host`) and matches the existing dashboards naming pattern. The `conduit-` prefix would also push every Conduit install across many profiles into a visually-clustered set of datasets in Honeycomb's UI, which sounds nice but isn't actually what an operator looking at the macOS board wants.
  - **A single `conduit` dataset for all profiles, with `profile.mode` as a resource attribute slicing within it.** Rejected because boards become harder to author (every panel needs a `profile.mode = X` filter, easy to forget), and Honeycomb dataset-scoped features (schemas, derived columns, triggers) get less useful when one dataset spans wildly different telemetry shapes (host metrics vs. windows event log vs. k8s pod metadata). Profile-distinct datasets keep each platform feeling first-class, matching the per-profile narrative the dashboards already establish.
  - **`action: upsert` for `service.name` with a separate "agent identity" attribute (`conduit.service.name`).** Rejected because it splits the concept: every consumer (boards, queries, alerts, runbooks) would need to know which attribute to filter on, and the OTel SemConv stance is that `service.name` is the canonical identity. The `insert` rule is the simpler and more honest answer.
  - **Pipeline-scoped resource processors (separate processors for OTLP-forwarded vs. host-emitted pipelines).** Rejected as overkill for V0. It's the more correct design — agent identity only on host pipelines, never on forwarded pipelines — but `action: insert` covers the pragmatic case (apps that emit `service.name` keep theirs; signals that lack one get filled) without doubling the processor count. Reconsider if a real use case for "force-overwrite incoming `service.name` on forwarded pipelines" emerges.
- **Consequences**:
  - **Schema / defaults work.** `internal/config/types.go` adds two helpers: `(*Profile).ResolvedPlatform() string` (mirrors the expander's `resolvePlatform()` so applyDefaults can share the same answer without a circular import) and `DefaultServiceNameForPlatform(platform string) string`. `internal/config/load.go`'s `applyDefaults()` calls them to fill `ServiceName` when empty. The validator in `internal/config/validate.go` keeps the existing "service_name required, non-empty string" rule, which now passes after defaults are applied for profiled installs and still fails for `mode: none` without explicit `service_name`.
  - **Expander work.** `internal/expander/templates/base.yaml.tmpl` flips `action: upsert` → `action: insert` for `service.name` only. `deployment.environment` is unchanged. All 10 goldens under `internal/expander/testdata/goldens/` regenerate to reflect the action change.
  - **Tests.** `internal/config/config_test.go` adds five tests covering profile-default fill-in (linux / darwin / windows / docker / k8s), an explicit-wins test, and a `mode: none` still-required test. `internal/expander/expander_test.go` adds an assertion that the rendered template has `action: insert` for `service.name`.
  - **Shipped configs work.** `deploy/{linux,windows,docker}/conduit.yaml.default` drop the `service_name: ${env:CONDUIT_SERVICE_NAME}` line entirely. The shipped `conduit.env.default` files drop `CONDUIT_SERVICE_NAME=` from the seeded variables — `HONEYCOMB_API_KEY` and `CONDUIT_DEPLOYMENT_ENVIRONMENT` remain the only seeded keys. Comment blocks in the `conduit.yaml.default` files document the profile default and the override path (literal `service_name: my-app` or hand-rolled `${env:CONDUIT_SERVICE_NAME}` for operators who want env-var indirection).
  - **Install script work.** `scripts/install_linux.sh` no longer seeds `CONDUIT_SERVICE_NAME` in `conduit.env`. When `--service-name=foo` is passed (or the interactive prompt receives a value), the script writes `service_name: foo` literally into `/etc/conduit/conduit.yaml` — idempotent (replaces an existing `^service_name:` line, otherwise appends). When no `--service-name` is passed and the operator accepts the interactive default, the script touches nothing — `applyDefaults()` fills in `linux-host` at agent startup. The interactive prompt's default value changes from `hostname -s` to `linux-host` to match what `applyDefaults()` would do anyway.
  - **Helm chart work.** `deploy/helm/conduit-agent/values.yaml` documents `conduit.serviceName` as optional (defaults to `k8s-cluster`). `deploy/helm/conduit-agent/templates/configmap.yaml` drops the `required` Helm function and conditionally emits `service_name:` only when `conduit.serviceName` is non-empty.
  - **Boards work.** Five board JSONs update log-panel `dataset:` fields to the profile default. The intro markdown in each board references the new dataset name. `dashboards/README.md` documents the dataset assumption: every checked-in board assumes the agent emits with the platform-default `service.name` unless the operator overrides.
  - **Operational risk: existing v0.0.2 installs.** Operators who installed v0.0.2 with `--service-name=foo` have `CONDUIT_SERVICE_NAME=foo` in `conduit.env` and `service_name: ${env:CONDUIT_SERVICE_NAME}` in `conduit.yaml`. On upgrade to a release that includes this ADR's changes, the package's `noreplace` semantics keep both files as the operator left them — the env-var resolution path keeps working at the Collector layer, and the new `applyDefaults()` doesn't touch a field that's already non-empty. New v0.0.3+ installs flow through the literal-write code path. No migration script is required.
  - **Operational risk: behavior change for forwarded OTLP traffic.** Operators relying on the existing `action: upsert` to overwrite an app's `service.name` (uncommon but possible) need to migrate to `transform/traces` or `resource/traces` with explicit upsert if they want that behavior. Release notes call this out; the doctor catalog gains a note in CDT0202 documentation pointing at this ADR. The actual incidence of "forwarded OTLP traffic that wants its `service.name` rewritten" is expected to be near zero — this is correcting a default, not removing capability.
