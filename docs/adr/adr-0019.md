# ADR-0019 — Container image published to ghcr.io/conduit-obs/conduit-agent

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Tech Lead, Field Engineering Lead (M4)
- **Context**: M4 ships Conduit's V0 Docker path. The milestone plan (and AC-04.1 in `01-product-requirements.md`) had used `honeycombio/conduit` as a placeholder image name with the actual registry venue gated on OQ-5. OQ-5's broader question (cosign / Authenticode / GPG signing infrastructure ownership) is unrelated to the registry-name choice and remains open; conflating them was blocking M4.C (multi-arch image publishing) on a decision that did not need to wait. Two registry venues were under consideration: Docker Hub under `honeycombio/conduit` (matching the placeholder) and GitHub Container Registry under the project's own `conduit-obs` org.
- **Decision**: Conduit's container image is published to `ghcr.io/conduit-obs/conduit-agent`. The repository name carries the `-agent` suffix to leave clean room for sibling images (`conduit-receiver`, etc.) under the same org if V2+ ever needs them. Tag scheme is the goreleaser default — `:vX.Y.Z`, `:vX.Y`, `:latest` — plus per-arch variants (`:vX.Y.Z-amd64`, `:vX.Y.Z-arm64`) that the multi-arch manifest references. The Helm chart, when M5 lands, ships to `ghcr.io/conduit-obs/charts/conduit-agent` (same org, same registry; OQ-2 still chooses ArtifactHub listing separately).
- **Alternatives considered**:
  - **Docker Hub `honeycombio/conduit`** (the M4 plan's placeholder). Rejected for V0 because (a) Docker Hub's free-tier rate-limit story is a known operational papercut for high-velocity deployments, and (b) sharing the `honeycombio` namespace conflates Conduit with Honeycomb's first-party tooling, which the positioning ([ADR-0007](adr-0007.md)) intentionally separates.
  - **`ghcr.io/conduit-obs/conduit`** (no `-agent` suffix). Rejected because V2+ may publish additional images under the same org; reserving the suffix scheme up front is cheaper than renaming later.
  - **AWS ECR Public, Quay, GitLab Container Registry, etc.** Rejected: ghcr.io is the natural pair for the GitHub-hosted source repo, integrates cleanly with goreleaser and GITHUB_TOKEN at release time, and matches the recommendation already in OQ-2 for the Helm chart.
- **Consequences**:
  - M4.C wires `dockers:` and `docker_manifests:` blocks in [`.goreleaser.yaml`](../../.goreleaser.yaml) targeting `ghcr.io/conduit-obs/conduit-agent`. CI publishes via the standard `docker/login-action` flow with `GITHUB_TOKEN`.
  - The `honeycombio/conduit` references in `01-product-requirements.md` §AC-04.1, `04-milestone-plan.md` §M4, `06-work-breakdown-structure.md` STORY-5.1, [`README.md`](../../README.md), and the [`deploy/docker/`](../../deploy/docker/) artifacts (Dockerfile build commands, compose example, in-image default config) are updated to `ghcr.io/conduit-obs/conduit-agent`.
  - OQ-5 is narrowed to signing infrastructure only (cosign / Authenticode / GPG / Helm provenance — all real questions, just orthogonal to registry venue).
  - The image-signing question (OQ-5) and the Helm chart hosting / ArtifactHub listing question (OQ-2) remain open; this ADR resolves the registry name only.
