# ADR-0005 — Generate RED metrics before any sampling step

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Tech Lead, Field Engineering Lead (V0 planning)
- **Context**: Operational dashboards (request rate, error rate, latency) must remain trustworthy regardless of trace sampling. Refinery is the canonical Honeycomb tail-sampler; if RED metrics are derived after sampling, dashboards lie.
- **Decision**: `spanmetricsconnector` is placed before any branch that could sample traces, in particular before forwarding to Refinery. RED metrics emit to Honeycomb at 100% even when traces are sampled.
- **Alternatives considered**:
  - Generate RED downstream of Refinery (rejected: dashboards become statistically wrong as sample rate drops).
  - Skip RED entirely and rely on Honeycomb-side derivation from sampled traces (rejected: same correctness problem).
- **Consequences**: RED metrics may include attributes from spans that were ultimately sampled out. This is correct behavior; the metrics describe the population, the traces are a sample of it. Customers must understand that RED counts and sampled-trace counts will differ — this is documented in the demo script ([`conduit-agent-plan/11-demo-and-poc-script.md`](../../conduit-agent-plan/11-demo-and-poc-script.md) §6) and in the Honeycomb-side investigation flow.
