# ADR-0016 — `memorylimiter` first, `batch` last, persistent queue on, redaction on

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Maintainer, Tech Lead (V0 planning)
- **Context**: Four common collector-deployment failure modes — OOM under traffic spikes, dropped data on egress outage, leaked secrets in logs, and resource starvation under burst — each have a known good upstream-component mitigation. Customers should not have to discover these defaults the hard way.
- **Decision**: Every Conduit pipeline is shaped with these defaults:
  1. `memorylimiter` is the first processor in every pipeline.
  2. `batch` is the last processor before the exporter.
  3. The egress exporter has a persistent queue backed by the `filestorage` extension, on by default.
  4. The log pipeline has `redactionprocessor` on by default with a conservative built-in pattern set (credit-card-shaped, AWS-key-shaped, JWT-shaped, basic-auth header).
- **Alternatives considered**:
  - Make any of the four off-by-default and let customers opt in (rejected: footgun-by-default).
  - Hardcode placement order without enforcement (rejected: golden-file tests in M2/M9 enforce placement, but the rule is easier to state as a default).
- **Consequences**: Slightly higher idle resource use (queue file, redaction overhead). Worth it. Documented in [`conduit-agent-plan/05-acceptance-criteria.md`](../../conduit-agent-plan/05-acceptance-criteria.md) under T-2.1, T-2.2, T-2.3, T-2.9.
