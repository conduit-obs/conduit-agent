# ADR-0012 — `conduit.yaml` expands to upstream collector YAML; `overrides:` is the only escape hatch

- **Status**: Accepted
- **Date**: 2026-05-01
- **Deciders**: Tech Lead, PM (V0 planning)
- **Context**: Two extremes are bad: a sealed black-box config (frustrates advanced users who legitimately need to reach upstream collector knobs) and a "just edit the OTel YAML" model (defeats the entire purpose of Conduit and recreates the familiarity gap). One escape hatch, scoped narrowly, is enough.
- **Decision**: User configuration is a high-level `conduit.yaml`. The agent expands it to upstream OpenTelemetry Collector YAML at runtime via embedded YAML fragments rendered through `text/template`, then handed to the collector resolver via the standard inline `yaml:` URI. An `overrides:` block lets advanced users inject raw collector YAML; `conduit doctor` warns when `overrides:` is non-empty. There is no `omit_base_components` knob.
- **Alternatives considered**:
  - Sealed config with no escape hatch (rejected: blocks legitimate advanced users).
  - Multiple overlap-prone escape hatches (`omit_base_components`, `extend_base`, etc.) (rejected: see [`conduit-agent-plan/12-observe-reference-notes.md`](../../conduit-agent-plan/12-observe-reference-notes.md) §"Anti-pattern: omit_base_components"). One hatch keeps the abstraction tight.
  - Raw collector YAML as the user surface (rejected: defeats the project's purpose).
- **Consequences**: Some users will live in `overrides:`. We treat that as a signal that the schema is missing something, not as a normal use case. Field engineers and PMs review heavy `overrides:` patterns at retro time and decide whether to promote them to first-class schema fields.
