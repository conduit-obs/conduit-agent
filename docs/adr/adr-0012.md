# ADR-0012 — `conduit.yaml` expands to upstream collector YAML; `overrides:` is the only escape hatch

- **Status**: Implemented (V0)
- **Date**: 2026-05-01 (decision); 2026-05-02 (implementation)
- **Deciders**: Tech Lead, PM (V0 planning)
- **Context**: Two extremes are bad: a sealed black-box config (frustrates advanced users who legitimately need to reach upstream collector knobs) and a "just edit the OTel YAML" model (defeats the entire purpose of Conduit and recreates the familiarity gap). One escape hatch, scoped narrowly, is enough.
- **Decision**: User configuration is a high-level `conduit.yaml`. The agent expands it to upstream OpenTelemetry Collector YAML at runtime via embedded YAML fragments rendered through `text/template`, then handed to the collector resolver as one or more inline `yaml:` URIs. An `overrides:` block at the root of `conduit.yaml` lets advanced users inject raw collector YAML; the expander emits the user's overrides as a *separate* second config source (not by merging it into the base render in Go), so the embedded collector deep-merges them at startup using its standard multi-config resolver semantics — maps merge by key (overrides win where they overlap), lists replace wholesale. `conduit doctor` will warn when `overrides:` is non-empty (M11). The validator catches typos at the top-level key (`receivers` / `processors` / `exporters` / `connectors` / `extensions` / `service`) so silent no-ops surface as errors. There is no `omit_base_components` knob.
- **Alternatives considered**:
  - Sealed config with no escape hatch (rejected: blocks legitimate advanced users).
  - Multiple overlap-prone escape hatches (`omit_base_components`, `extend_base`, etc.) (rejected: see `conduit-agent-plan/12-observe-reference-notes.md` §"Anti-pattern: omit_base_components"). One hatch keeps the abstraction tight.
  - Raw collector YAML as the user surface (rejected: defeats the project's purpose).
- **Consequences**: Some users will live in `overrides:`. We treat that as a signal that the schema is missing something, not as a normal use case. Field engineers and PMs review heavy `overrides:` patterns at retro time and decide whether to promote them to first-class schema fields. The implementation choice — emitting overrides as a second config URI rather than merging in Go — means we never need to maintain our own deep-merge code (the collector's resolver is canonical) and `conduit preview` shows the base + override documents separated by `---`, matching what `otelcol --config base.yaml --config overrides.yaml` would produce.
