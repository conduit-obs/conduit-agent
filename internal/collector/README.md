# internal/collector

Holds the embedded upstream OTel Collector module that ships in the Conduit binary. The contents of this directory are **fully generated** by the OpenTelemetry Collector Builder (OCB) from [`builder-config.yaml`](../../builder-config.yaml) at the repo root, then folded into the source tree under the root Go module so `go build ./...` works end-to-end.

Do **not** hand-edit anything in here. Every change goes through `builder-config.yaml`.

## Status

- **M2.A** (done): `builder-config.yaml` is authored; `make build-ocb` invokes OCB and writes raw output to `build/collector/`.
- **M2.B** (done): `internal/config/` defines the conduit.yaml schema, `internal/expander/` renders it to upstream OTel YAML, `conduit config --validate` and `conduit preview` are wired.
- **M2.C** (done): `make build-ocb` folds OCB output (only `components.go` for now) into this directory with the package rewritten to `package collector`. `collector.go` (hand-written) exposes `DefaultSettings(buildInfo)` and `Run(ctx, settings)` so `cmd/run` boots the embedded collector by feeding it the YAML from `internal/expander`.

Files in this directory:

- `collector.go` — hand-written. Owns `DefaultBuildInfo`, `DefaultSettings`, and `Run`. Edit freely; this is the integration contract.
- `components.go` — generated by OCB. Do not edit; re-run `make build-ocb` after changing `builder-config.yaml`.
- `README.md` — this file.

Files OCB also generates but Conduit currently drops:

- `main.go` — replaced by `collector.go`.
- `main_others.go` / `main_windows.go` — Windows-service launcher path. Re-introduced in M4 alongside MSI packaging.
- `go.mod` / `go.sum` — Conduit uses a single root module; the deps OCB pulls land in the root `go.mod` via `go mod tidy`.

## Build flow

```bash
make install-ocb        # one-time download of the pinned OCB binary
make build-ocb          # regenerate this directory from builder-config.yaml
make build              # go build ./bin/conduit
```

`make build` does not depend on `make build-ocb`; the generated code is committed to git so a fresh clone can `go build` directly. Re-run `make build-ocb` whenever `builder-config.yaml` changes (adding or removing a component, bumping the upstream version).

## Why we go this route

See [`docs/adr/adr-0003.md`](../../docs/adr/adr-0003.md) (OCB as the build engine) and [`docs/adr/adr-0004.md`](../../docs/adr/adr-0004.md) (pure-upstream components only). The architectural rationale (component manifest, lifecycle, version pinning, CVE response) is in [`conduit-agent-plan/03-technical-architecture-v0.md`](../../conduit-agent-plan/03-technical-architecture-v0.md) §"OCB-based distribution".
